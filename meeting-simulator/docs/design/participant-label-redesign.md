# 参会者标签重设计方案

## 问题诊断

当前 `roles` 只有 `name / title / type(leader|collaborator|challenger)`，前端只展示名字和职位。`type` 用于内部配色，用户看不到。

用户反馈：性格标签（"追问细节""掌控节奏"）太抽象，不像真实开会。

**根因分析**：开会前你真正想知道的不是"这人什么性格"，而是：

1. **这个人跟我是什么关系？** 直属上级？平级？跨部门？
2. **他在这件事上什么立场？** 支持我？反对我？中立观望？
3. **我该对他有什么心理准备？** 会帮我还是会怼我？

性格是长期特质，但会议是一个具体博弈场景。同一个人，在不同议题上可以是你的盟友，也可以是你的对手。**立场 > 性格**。

---

## 方案 A：一句话关系速写（推荐）

**核心思路**：不用标签，用一句自然语言描述"这个人和你在这场会中的关系"。像同事在会前跟你耳语。

### 数据结构变更

```json
{
  "name": "Daniel Chen",
  "title": "Engineering Lead",
  "avatar": "DC",
  "type": "challenger",
  "briefNote": "你的直属老板，最近对项目进度不太满意",
  "stance": "pressure"
}
```

新增字段：
- `briefNote`（string）：一句话关系速写，15-25 个中文字，用"你"作为主语锚点
- `stance`（enum）：`ally | neutral | pressure`，用于前端配色/图标

### 前端展示示例

```
[头像] Daniel
       Engineering Lead
       "你的直属老板，最近对项目进度不太满意"
```

```
[头像] Mia
       Product Manager
       "跨部门同事，这个需求是她推的，会替你说话"
```

```
[头像] Ryan
       QA Lead
       "上次就质疑过你的排期，这次大概率还会追"
```

### 优点
- 信息密度高，一句话传递关系+立场+心理预期
- 最接近真实职场心理："开会前同事跟你八卦"
- 自然语言比任何标签都灵活

### 缺点
- 对 LLM 生成质量要求高，需要在 prompt 中精确约束

---

## 方案 B：立场标签 + 关注点

**核心思路**：用两个短标签替代性格标签。第一个标签描述立场（跟你的关系），第二个标签描述他在本场会中关注什么。

### 数据结构变更

```json
{
  "name": "Daniel Chen",
  "title": "Engineering Lead",
  "avatar": "DC",
  "type": "challenger",
  "stanceTag": "你的直属上级",
  "focusTag": "关注交付时间"
}
```

新增字段：
- `stanceTag`（string）：关系标签，5-8 个字。限定词表：你的直属上级 / 平级同事 / 跨部门 / 项目盟友 / 利益相关方
- `focusTag`（string）：关注点标签，4-6 个字。如：关注交付时间 / 盯预算 / 推需求 / 关注质量

### 前端展示示例

```
[头像] Daniel
       Engineering Lead
       [你的直属上级]  [关注交付时间]
```

```
[头像] Mia
       Product Manager
       [项目盟友]  [推需求落地]
```

```
[头像] Ryan
       QA Lead
       [跨部门同事]  [盯测试覆盖]
```

### 优点
- 结构化，前端排版稳定，扫一眼就懂
- 标签可以做配色（盟友=绿，压力=橙，中立=灰）

### 缺点
- 信息密度不如方案 A，少了"历史恩怨"这类细节
- 标签词表需要维护，过于模板化可能又回到"太抽象"的老问题

---

## 方案 C：会前情报卡（信息量最大）

**核心思路**：不只是标签，而是给每个参会者一张迷你情报卡，包含关系、立场、历史记录。像 RPG 游戏里查看 NPC 信息面板。

### 数据结构变更

```json
{
  "name": "Daniel Chen",
  "title": "Engineering Lead",
  "avatar": "DC",
  "type": "challenger",
  "stance": "pressure",
  "intel": {
    "relationship": "你的直属老板",
    "attitude": "对你目前的进度有疑虑",
    "history": "上周周会追问过 API 延期的事"
  }
}
```

### 前端展示示例

默认态：跟方案 A 一样，只显示名字 + 职位 + 一句话。

点击展开（或长按）：

```
Daniel Chen · Engineering Lead
------------------------------
关系：你的直属老板
态度：对你目前的进度有疑虑
上次：上周周会追问过 API 延期的事
```

### 优点
- 信息最丰富，沉浸感最强
- 适合想深度准备的用户

### 缺点
- 信息量大，会前页面可能过重
- LLM 需要生成更多内容，token 成本上升
- 对于快速开始的用户是负担

---

## 决策建议

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 信息密度 | 高 | 中 | 最高 |
| 开发成本 | 低 | 低 | 中 |
| LLM 生成难度 | 中 | 低 | 高 |
| 真实感 | 最强 | 一般 | 强 |
| 扫一眼就懂 | 是 | 是 | 需要交互 |

**推荐方案 A（一句话关系速写）**，理由：

1. **最接近真实心理模型**。开会前你想知道的就是"这人跟我啥关系、会不会怼我"，一句自然语言最直接
2. **实现成本低**。只需在 prompt 中加约束 + 前端加一行文字
3. **灵活度高**。自然语言能表达标签无法覆盖的微妙关系（"上次帮过你""她跟你老板关系好"）
4. `stance` 字段（ally/neutral/pressure）保留用于配色，替代当前的 `type` 配色逻辑

### 下一步

1. **架构师评审**：确认数据结构变更对后端/数据库的影响
2. **Prompt 工程**：在 `generate-meeting.js` 中增加 `briefNote` 和 `stance` 的生成约束
3. **前端实现**：修改 `ParticipantCard` 组件，展示 `briefNote`
4. **测试验证**：生成 5-10 场会议，检查 `briefNote` 质量是否稳定
