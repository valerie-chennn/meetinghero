# MeetingHero 产品方向重新定位方案

**文档版本**: v1.0  
**作者**: Product Manager  
**日期**: 2026-04-01  
**状态**: 待架构师/设计师评审

---

## 1. 产品重新定位

**一句话定义**：用你喜欢的故事练英语开口，说话才是主角。

> 从"职场英语培训工具"转变为"沉浸式英语开口引擎"——学习目标不变，但驱动力从义务感换成了好奇心和代入感。

---

## 2. 为什么要转变（问题根因）

**现有产品的心理门槛分析**（基于代码逻辑推断，需用户访谈验证）：

| 门槛类型 | 具体表现 | 来源 |
|---------|---------|------|
| 情景门槛 | 只有职场周会，场景单一，没有情绪代入 | `generate-meeting.js` 硬编码 "Weekly Project Sync" |
| 身份门槛 | 用户必须以自己的职位/行业角色代入，暴露真实痛点 | `AppContext` 持久化 `jobTitle / industry` |
| 压力门槛 | NPC 含 challenger（施压者），会议本身就是职场焦虑的来源 | 系统 prompt 要求"至少 1 个 pressure 角色" |
| 语言门槛 | 真实职场语境要求精准，用户怕说错 | 复盘机制会逐句分析 betterWay，强化了"说错了"的感知 |

**转变逻辑**：降低前三个门槛（情景、身份、压力），同时保留语言门槛作为学习价值的来源。

---

## 3. 四种场景系统设计

### 3.1 场景一：职场深度线（Evolution，原场景升级）

**定义**：保留职场周会壳子，但注入人际暗线——NPC 之间有私下恩怨、权力博弈和隐藏动机，会议不再是纯工作汇报，而是一出职场暗战。

**与现有产品的核心差异**：现在的 NPC 是扁平的"主持人/协作者/挑战者"功能角色；升级后每个 NPC 有背景故事和人际关系网，用户不只是汇报，而是在博弈中开口。

**用户价值**：
- 代入感强：会议里藏着戏，用户想看懂情节就得认真听
- 现实迁移性高：练的还是真实职场英语，能直接用
- 复购动力：每次生成的人际暗线不同，有重玩价值

**留存逻辑**：NPC 关系网解锁（第 N 次会议后揭露某 NPC 的隐藏动机），类似剧情解锁机制。

**信心等级**：确定（架构改动最小，风险最低）

**3 个具体示例**：

> 示例 A：市场总监 Ryan 这场突然发难，因为上次你的项目抢了他的预算。他的质疑不是技术问题，是政治信号——narrator 会告诉你这个，但你得自己决定怎么回应。

> 示例 B：你的老板 Sarah 在开会前刚刚被大老板批了，她今天格外挑剔。你的盟友 Leo 私下传话说"别在 Sarah 面前提延期，换个词"——narrator 在关键节点前会给你这个情报。

> 示例 C：两个 NPC（Emma 和 David）明面上是协作者，实际在抢同一个晋升名额。他们的发言方向相反，你的站队选择会影响他们对你的态度。

### 3.2 场景二：IP 剧情角色扮演（IP Mode）

**定义**：用户选择一个经典 IP（西游记、三国、哈利波特、星战等），系统将其代入某个经典情节的"会议版本"。所有对话用英语，情节有原著逻辑支撑，用户不需要编造背景。

**核心设计**：把情节中的冲突节点转化为"开口时刻"。唐僧小队要不要绕路、刘备要不要借荆州、赫敏要不要举报哈利——这些决策需要用英语表达立场。

**用户价值**：
- 零代入摩擦：用户已经熟悉这些角色和情节，不需要学习成本
- 情感驱动强：为孙悟空辩护比为 Sprint 6 辩护动力强得多
- 社交传播性高：截图"刚跟猪八戒用英语开了个会"本身就是内容

**留存逻辑**：IP 图鉴解锁（完成某 IP 的所有场景），情节分支（同一情节不同立场）。

**信心等级**：推测（IP 场景能否有效驱动开口行为，需要 A/B 测试验证；版权方面也需法务确认，使用公版 IP 或改编后的平行世界版本可规避）

**3 个具体示例**：

> 示例 A：西游记 x 取经路线规划会。孙悟空提议直接飞过去，猪八戒说腿酸要走大路，沙僧一言不发。唐僧（用户）用英语主持：你如何回应悟空"fly directly"的提议并给出理由？

> 示例 B：哈利波特 x 魔法部危机应对会。乌姆里奇刚宣布新规定，邓布利多校长要求各院长汇报应对方案。用户扮演麦格教授，用英语向校长汇报格兰芬多的应对策略，同时顶住斯内普的质疑。

> 示例 C：三国 x 赤壁战前动员会。周瑜需要说服众将领支持火攻。用户扮演鲁肃，英语发言支持周瑜方案，同时回应张昭的质疑。注：角色说中文是历史事实，但这里设定为"平行宇宙英语版"。

### 3.3 场景三：大乱炖（Chaos Mode）

**定义**：系统随机（或用户手选）拼接不同世界的角色参加同一场会议，主题也可以荒诞。乔布斯 + 猪八戒 + 林黛玉讨论"iPhone 要不要加折叠屏"，用户扮演会议主持或某个角色。

**核心设计亮点**：错位本身就是笑点和话题，不需要用户提前学习任何背景知识，完全可以乱说——但得用英语说。

**用户价值**：
- 心理门槛最低：本来就是乱的，说错了没负担
- 社交传播系数最高：内容天然有梗，用户愿意截图发群
- 创作感：用户可以手动选角色组合，有 DIY 乐趣

**留存逻辑**：解锁新角色池（完成 X 场后解锁新历史/虚构角色），每日随机乱炖推送。

**信心等级**：推测（荒诞场景能否维持足够的语言学习价值存疑。如果太好笑，用户可能只享受荒诞感而不认真输出英语。需要在玩法设计上加入约束：比如每个关键节点的语言难度和评分标准与其他模式相同）

**3 个具体示例**：

> 示例 A：乔布斯 + 孔子 + 马斯克开发布会 review 会。乔布斯觉得产品不够 insanely great，孔子说要以德服人，马斯克说要上火星。用户扮演 CMO，用英语协调三方找到发布会主题。

> 示例 B：林黛玉 + 多拉A梦 + 李白 讨论年度预算分配。林黛玉要求为诗集追加预算，多拉A梦说明年要造时光机，李白坚持酒钱不能削减。用户扮演 CFO 用英语做最终裁决。

> 示例 C：诸葛亮 + 爱因斯坦 + 村上春树 谈新项目立项。诸葛亮要求项目要有天下大义，爱因斯坦说要做相对论应用，村上春树说项目名字必须忧郁。用户扮演 CEO 用英语给出项目方向。

### 3.4 场景四：抓马剧情（Drama Mode）

**定义**：高冲突剧情设定——真假千金董事会对决、宫斗权谋、豪门撕逼、职场复仇等。用户被推入一个"不说话就输"的紧张局面，驱动力是戏剧张力而非学习义务。

**核心设计**：每个关键节点都是一个戏剧高潮。比如竞争对手当众发难，沉默 3 秒就等于认怂——用户不说是剧情上的失败，而不是学习上的失败。把"语言焦虑"转化为"戏剧张力"。

**用户价值**：
- 开口驱动力最强：剧情设计让沉默有代价，用户不得不说
- 情绪峰值最高：冲突场景让人肾上腺素上升，开口阻力反而下降
- 完播率高：高冲突剧情用户会想"看到最后"

**留存逻辑**：剧情结局分支（用户的每次发言影响结局走向），解锁隐藏结局（某种回应方式触发特殊剧情）。

**信心等级**：推测（此场景对内容质量要求最高，剧情不够好用户会出戏；前期内容供给成本大，可能需要人工编辑参与，而非纯 AI 生成）

**3 个具体示例**：

> 示例 A：真假千金大战。你是原本的公司继承人，今天董事会上一个"失散多年的女儿"突然出现，当众宣称自己才是合法继承人。董事们都在观望。用英语，现在，反击。

> 示例 B：职场复仇局。三年前你被上司坑走了晋升机会，今天对方突然来你团队述职，你是评审委员之一。你可以用问题问倒他，也可以选择放水。现在他刚讲完，轮到你用英语提问了。

> 示例 C：宫斗版本。你是新入宫的贵人，今天的茶会上，大宫女当众用英语"礼貌地"暗讽你出身低微。所有人都在看你怎么接。沉默 = 认输。你要用英语，温柔而致命地回击。

## 4. 对现有架构的影响分析

### 4.1 可以直接复用的模块（0 改动）

| 模块 | 文件位置 | 复用说明 |
|------|---------|---------|
| 会中对话流引擎 | `routes/meeting.js` `/respond` 接口 | 用户发言 → AI 响应的核心逻辑完全通用，与场景无关 |
| 语音输入 | 前端音频录入组件 | 所有场景都需要语音输入，直接复用 |
| 复盘系统 | `prompts/generate-review.js` | 语言分析逻辑（betterWay、pattern、practice）与场景无关，可复用 |
| 关键节点交互框架 | `isKeyNode` / `nodeIndex` / `nodeType` 结构 | 三节点结构适用于所有场景，只需替换内容 |
| 英语等级分层 | `getReferenceStrategy` / `getNpcLengthConstraint` | 完全复用，参考说法和 NPC 长度约束与场景无关 |
| 全局状态管理 | `AppContext.jsx` | `sessionId / englishLevel / conversations` 等字段与场景无关 |
| 数据库 schema | `meetings` / `conversations` 表结构 | 结构通用，无需改动 |

### 4.2 需要改造的模块

#### 来源选择页（SourceSelect.jsx）— 改动幅度：大

**现状**：只有两个选项（生成模拟会议 / 上传材料），是线性流程。

**需要变成**：场景选择入口，用户在这里选择 4 种模式之一。每种模式有不同的子流程（IP 模式需要选 IP，大乱炖需要选角色组合）。

**改动点**：
- 组件从 2 选项列表改为 4 种场景卡片
- 增加每种场景的二级选择流程（IP 选择、角色选择等）
- `meetingSource` 字段从 `generate/upload` 扩展为 `workplace/ip/chaos/drama`

#### 会议生成 Prompt（generate-meeting.js）— 改动幅度：大

**现状**：硬编码为 "Weekly Project Sync"，NPC 都是职场角色，叙事逻辑是职场会议。

**需要变成**：基于场景类型（sceneType）动态切换 systemPrompt 模板。

**改动点**：
- `generateMeetingPrompt` 增加 `sceneType` 参数
- 为每种场景编写专属 systemPrompt（NPC 定义、叙事逻辑、关键节点类型）
- narrator 的"情报"类型需随场景调整（职场暗线 vs 剧情背景 vs 混乱喜剧）
- IP 模式需要额外的 `ipName + sceneTitle` 参数
- 大乱炖模式需要 `roleList[]` 参数（用户手选角色）

**注意**：复盘 prompt 可以基本不改，因为语言分析逻辑与场景无关；但 `achievement / improvement` 的语气措辞可以考虑随场景变化（抓马模式复盘语气可以更"戏剧化"）。

#### 角色系统（NPC 结构）— 改动幅度：中

**现状**：NPC 有 `type: leader/collaborator/challenger` 和 `stance: ally/neutral/pressure`，是职场功能角色定义。

**需要变成**：不同场景的 NPC 需要不同的角色定义维度。
- IP 模式：角色来自原著，需要 `ipCharacter / originalIp` 字段，`stance` 仍然适用
- 大乱炖：角色来自不同世界，需要 `world / era` 字段标注来源，增加喜剧冲突感
- 抓马模式：需要 `relationship` 字段（竞争对手、盟友、中立观察者），以及 `secretAgenda` 隐藏动机

**关键决策**：是否扩展现有 NPC schema，还是为不同场景维护不同 schema？

**建议**：扩展现有 schema，新字段都设为可选（optional）。这样复用成本最低，但需要前端做好字段空值处理。代价是 schema 会变复杂，维护成本上升。

#### `AppContext.jsx` — 改动幅度：小

**现状**：持久化字段为 `sessionId / userName / englishLevel / jobTitle / industry`。

**需要变成**：
- 增加 `sceneType`（当前选择的场景类型）
- IP 模式增加 `selectedIp` / `selectedScene`
- 大乱炖增加 `selectedRoles[]`

`jobTitle / industry` 对于 IP 模式和大乱炖没有意义，但保留不影响功能；Onboarding 流程可以设为"职场场景才必填"。

### 4.3 新增模块

| 模块 | 必要性 | 说明 |
|------|-------|------|
| IP 内容库 | 必须 | 预定义 IP 列表 + 每个 IP 的场景列表（标题、情节背景、角色列表）。前期可以做 6-10 个 IP，每个 2-3 个场景 |
| 角色池（大乱炖用） | 必须 | 预定义可选角色列表，含角色名、来源 IP/历史背景、个性标签，供用户选择 |
| 场景卡片 UI 组件 | 必须 | 4 种场景的视觉化选择入口，需要设计规范（每种场景有独特视觉风格）|
| 剧情结局分支系统 | 可选 | 抓马模式的结局差异化，技术复杂度中等，MVP 可以先做单一结局 |
| 每日推荐/随机乱炖生成 | 可选 | 大乱炖的留存机制，MVP 可以先做用户手动选择 |

## 5. MVP 优先级建议

### 推荐 MVP：职场深度线（Evolution）+ 抓马剧情（Drama）并行上线

**理由和取舍分析**：

#### 为什么先做职场深度线

**做了获得的**：
- 改动量最小，基于现有代码迭代，风险最低
- 直接解决用户反馈的"太职场、压力大"问题，而不是换赛道
- 现有用户（已经接受职场场景的人）留存率提升最直接
- 可以作为技术验证：新的 NPC 关系网设计是否能让会议体验更有趣

**放弃的**：
- 对于不喜欢职场场景的潜在新用户，这个改动吸引力有限
- 人际暗线的内容质量高度依赖 AI 生成质量，可能需要较多调优

**验收标准（上线后 2 周内）**：用户完成率（完成 3 个关键节点的比例）比原版提升 15%+，NPS 中"有趣/有代入感"相关反馈增加。

#### 为什么先做抓马剧情（而不是 IP 或大乱炖）

**做了获得的**：
- 开口驱动力最强，直接解决核心问题（降低说话门槛）
- 内容上不依赖版权 IP，不需要维护庞大内容库
- 场景天然有传播性，用户愿意主动分享截图（获客价值）
- 验证"戏剧张力驱动开口"这一核心假设的成本最低

**放弃的**：
- 剧情内容质量要求高，纯 AI 生成可能出戏，前期可能需要人工编辑部分剧本
- 抓马场景的英语学习价值感知可能比职场场景弱，需要复盘系统强化学习感

**验收标准（上线后 2 周内）**：新注册用户中选择抓马场景的比例 > 40%，社交分享率（会话结束后的分享行为）比职场模式高 2 倍+。

#### 为什么 IP 模式和大乱炖放到第二期

**IP 模式推迟的代价**：
- 错过最容易做出差异化的场景，竞品可能在这段时间跟进
- 潜在的高传播性内容延迟上线

**IP 模式推迟的理由**：
- 需要构建内容库（IP 列表 + 场景库），前期工作量大，属于"内容工程"而非"功能工程"
- 版权确认需要时间（公版 IP 没问题，但商业 IP 如哈利波特需要评估风险）
- 场景质量参差不齐（西游记写得好，不熟悉的 IP 可能踩坑），需要内容审核机制

**大乱炖推迟的代价**：
- 最有传播性的玩法延迟上线，错过社交裂变窗口

**大乱炖推迟的理由**：
- 产品假设未验证：荒诞场景能否维持语言学习价值，用户是否会认真说话还是只享受荒诞感
- 角色池建设需要时间（需要整理足够多的角色让用户选择）
- 建议先通过抓马模式验证"非职场场景驱动开口"的核心假设，再上大乱炖

### MVP 上线优先级排序

| 优先级 | 场景 | 核心验证假设 | 估计开发周期 |
|--------|------|------------|------------|
| P0 | 职场深度线 Evolution | 人际暗线能提升会议代入感 | 2-3 周 |
| P0 | 抓马剧情 Drama | 戏剧张力能降低开口门槛 | 3-4 周（需内容设计）|
| P1 | IP 剧情角色扮演 | IP 熟悉感能消除情景门槛 | 6-8 周（含内容库建设）|
| P2 | 大乱炖 Chaos | 荒诞感能消除语言焦虑 | 4-5 周（含角色池建设）|

> **估计信心等级**：推测。以上时间估算仅基于架构阅读，未经工程师评审，请架构师确认。

## 6. 关键风险与约束

| 风险 | 严重程度 | 应对方式 |
|------|---------|---------|
| 抓马/IP 场景的英语学习价值感知下降 | 高 | 复盘系统不变（保持语言分析深度），并在场景结尾强化"这场会你学到了什么"的感知 |
| AI 生成的人际暗线/剧情质量不稳定 | 中 | 前期对高优场景（抓马、职场暗线）进行人工校验，建立 prompt 质量检查清单 |
| IP 版权风险 | 中 | MVP 只使用公版 IP（西游记、三国、莎士比亚等），商业 IP 进入第二期前法务评审 |
| 大乱炖用户只享乐不学习 | 中 | 关键节点的评分标准和语言要求不降级，用喜剧包装但学习深度不变 |
| 场景增多导致 Onboarding 流程复杂 | 低 | 重新设计 Onboarding：先选场景类型，再填必要信息（职场场景才需要 jobTitle/industry）|

### 同意的部分

1. **产品重新定位方向正确** — "用你喜欢的故事练英语开口"清晰有力

4. **四种场景的示例具体**，看完就知道体验是什么样的

### 不同意的部分

#### MVP 优先级调整

产品经理建议：职场深度线（P0）+ 抓马剧情（P0），大乱炖和 IP 放 P1/P2。

**Leader 建议调整为：大乱炖（P0）+ 抓马剧情（P0），职场深度线降为 P1。**

理由：

| 对比维度 | 职场深度线 | 大乱炖 |
|---------|-----------|-------|
| 用户兴奋度 | 低（还是开会） | 高（林黛玉×乔布斯开会） |
| 实现成本 | 中（NPC 关系网设计） | 低（随机角色+随机主题，改 prompt 就行） |
| 传播性 | 弱 | 极强（天然社交货币） |
| 验证价值 | 验证"暗线能否提升代入感" | 验证"非职场场景能否驱动开口" |

职场深度线本质是现有模式的增强，改不改用户感知差异不大。不如把精力放在真正让人眼前一亮的新模式上。

#### 方案中的不足

1. **场景之间的关系没理清** — 大乱炖和抓马剧情在实现上高度重叠（都是非职场角色 + 戏剧冲突），可以共享一套 prompt 模板，只是参数不同（角色来源、冲突强度、叙事风格）。不应该当四个独立模块设计。

2. **没敢砍需求** — 四个方向全接了，没问"有没有伪需求"。比如 IP 模式的内容库建设成本高，当前阶段应该明确说"先不碰"而不是排到 P1 留着。

3. **prompt 改造评估太粗** — 核心就是 `generateMeetingPrompt` 加 `sceneType` 分支，每个分支一套角色定义和叙事规则。应该给出具体的改造方案而不是只说"改动幅度大"。

4. **时间估算没有依据** — "3-4 周""6-8 周"是猜的，不如不给。应该等架构师评审后再出。

### Leader 建议的 MVP 路线

```
第一期（立即开始）：
├── 大乱炖模式 — 随机角色 + 随机主题，改 prompt + 场景选择页
├── 抓马剧情模式 — 高冲突情节模板，改 prompt + 场景选择页
└── 共享改造：sceneType 参数贯穿全链路、SourceSelect 页重做为场景选择

第二期（第一期验证后）：
├── 职场深度线 — NPC 关系网、人际暗线
└── IP 模式 — 内容库建设、角色图鉴

关键原则：大乱炖和抓马共享一套 prompt 模板基础设施，不重复建设。
```
