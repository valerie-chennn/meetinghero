## 前置说明：三条约束的影响

老板的反馈隐含了三条硬约束，贯穿所有方向：

1. **职场暗线砍掉**：四个方向变三个方向（IP+大乱炖合并、抓马、正常开会）
2. **IP 靠 web search 实时获取**：不需要维护内容库，但需要设计搜索策略和信息提取逻辑
3. **花名保留**：所有模式的 Onboarding 第一步都是取花名

这些约束直接影响优先级排序和工程量评估，分析时会逐一标注。

## 方向一：IP + 大乱炖合并方案

### 1.1 核心设计决策：两者的关系

IP 模式和大乱炖合并后，本质是同一套机制的两个参数版本：

- **IP 模式**：用户指定一个 IP（西游记），系统从该 IP 中取角色，生成会议
- **大乱炖**：用户不指定（或系统随机），从多个不同 IP 各取 1-2 个角色，拼在一起

**不是两个独立功能，是同一个角色选择界面的两种使用路径。**

这意味着代码层面只需要一套实现：`selectedCharacters[]` 数组，每个元素带 `ipSource` 字段标记来源。

### 1.2 搜索策略设计

**问题**：web search 实时获取 IP 信息，搜什么？搜到的是非结构化内容，怎么转成角色卡？

**方案**：不直接搜"角色介绍"，而是让 AI 生成搜索关键词，再用搜索结果喂给 AI 做角色信息提取。

**搜索关键词模板（按优先级）**：

```
// 精确角色搜索（用户已知角色名）
"{characterName} {ipTitle} character personality traits"
"{characterName} {ipTitle} wiki"

// IP 角色列表搜索（用户只知道 IP 名，需要列出角色）
"{ipTitle} main characters list"
"{ipTitle} character relationships"

// 中文 IP 英文化搜索
"{ipTitle} English version characters"
"Journey to the West characters personalities"  // 西游记英文名
```

**实际搜索流程**：

```
第 1 步：用户输入"西游记"
第 2 步：后端用"Journey to the West main characters" 搜索
第 3 步：搜索结果（wiki 段落、角色介绍页）传给 AI
第 4 步：AI 从搜索结果提取结构化角色卡：
         { name, ipSource, personality[], speakingStyle, stance, briefNote }
第 5 步：前端展示角色卡，用户选择
```

**关键问题：搜索 API 用什么？**

选项对比：

| 方案 | 成本 | 速度 | 质量 |
|------|------|------|------|
| Bing Search API | 免费额度有限，超出 $3/1000次 | 快 | 高（权威源） |
| Serper API | $50/月起 | 快 | 高 |
| 直接让 AI 用训练知识生成 | 0 | 极快 | 中（知识截止日期问题，但西游记、三国这类 IP 完全够用） |

**推荐方案**：MVP 阶段用"AI 训练知识直接生成"，不调用外部搜索 API。原因：
- 西游记、三国、哈利波特等主流 IP，AI 知识库已经足够准确
- 节省搜索 API 成本和延迟（web search 加搜索 + 提取至少多 3-5 秒）
- 产品侧的价值在于"把 IP 信息转化为会议角色"，而不在于"用搜索实时抓取"
- 如果用户搜了一个冷门 IP，AI 知识不足时可以降级提示"换一个更知名的 IP"

**如果必须支持冷门 IP 或新番**，再接入搜索 API（Phase 2）。关键词策略已设计好，接入时直接用。

### 1.3 用户角色选择交互流程

**设计原则**：不要搜索框——用户通常不知道要搜什么。用分层浏览+快速推荐。

**完整交互流程**：

```
[场景选择页]
用户点击"IP 会议 / 大乱炖"卡片
         ↓
[IP 选择页 - 第一屏]
展示 IP 分类标签：
  「经典名著」西游记 / 三国演义 / 红楼梦
  「欧美奇幻」哈利波特 / 冰与火之歌 / 指环王
  「科幻」星际迷航 / 星球大战 / 三体
  「韩剧/华剧」（待扩展）
  「自定义」→ 输入框搜索

用户点击某个 IP（如"西游记"）
         ↓
[角色展示页 - 第二屏]
系统调用 AI 生成角色卡列表（约 2-3 秒 loading）
展示 4-6 个主要角色，每个角色卡包含：
  - 角色名（中英文）
  - 简短人设标签（如"急躁 / 忠诚 / 爱吹牛"）
  - 一句话说话风格（如"直接表态，说话简短有力"）
  - 来源 IP 标签

用户操作分支：
  路径 A（IP 模式）：点"用这个 IP 开会" → 系统从这个 IP 选 3 个 NPC
  路径 B（大乱炖）：点"选这个角色" → 可继续选其他 IP 的角色（最多选 3 个）
  路径 C（随机大乱炖）：点"随机乱炖" → 系统随机从 3 个不同 IP 各选 1 个角色
         ↓
[用户角色分配]（见方向四）
         ↓
[会议生成 Loading]
```

**大乱炖的"随机乱炖"按钮很重要**：降低选择成本，让犹豫的用户直接跳过选角色环节。

### 1.4 角色信息提取：从 AI 知识到角色卡

**Prompt 设计（角色卡生成）**：

```
输入：ipTitle = "西游记（Journey to the West）"

Prompt：请为 Journey to the West 生成 5-6 个主要角色的角色卡，每个角色包含：
- name（英文名）
- nameZh（中文名）
- ipSource（"Journey to the West"）
- personality（3-4个性格标签，中文）
- speakingStyle（一句话说话风格描述，中文，≤20字）
- stance（在会议场景中的默认立场：ally/neutral/pressure）
- briefNote（一句话角色介绍，中文，≤15字，用于卡片展示）
- meetingBehavior（在会议中的典型行为，中文，≤20字）

输出纯 JSON 数组。
```

**提取质量保证**：对于主流 IP（西游记、三国、哈利波特），AI 训练知识充足，角色信息准确。需要加一条后处理校验：如果生成的角色列表为空或字段不完整，降级到预置的兜底角色卡（每个大 IP 手工预置 4 个主要角色作为 fallback）。

**后备 fallback 策略**：

为 6-8 个最热门 IP 手写兜底角色卡（不多，总共约 40-50 个角色），存在代码里。AI 生成失败时直接用兜底卡。这是"零内容库"和"稳定性"之间的平衡点。

### 1.5 大乱炖的会议主题怎么定

大乱炖的荒诞感来源之一是"不搭的角色讨论不搭的话题"。会议主题的生成逻辑：

- **随机大乱炖**：系统从荒诞主题池中随机选（"讨论 iPhone 要不要加折叠屏" / "年度预算分配" / "新品牌代言人选谁"）
- **用户自选角色**：系统根据角色组合智能生成一个主题（孙悟空+爱因斯坦+林黛玉 → "如何在 30 天内完成不可能的任务"）
- **用户自定义**：允许用户在角色选好后手动输入会议主题（可选步骤，有跳过按钮）

### 1.7 风险与信心等级

| 风险 | 程度 | 应对 |
|------|------|------|
| AI 对冷门 IP 角色信息不准确 | 中 | 预置 fallback 卡片 + 提示用户换知名 IP |
| 角色卡生成需要额外 AI 调用（成本和延迟） | 低 | 角色卡生成约 1-2 秒，与主会议生成分开，体验上可接受 |
| 大乱炖用户只享受荒诞感不认真说话 | 中 | 关键节点评分标准不降级，用荒诞包装但语言要求不变 |
| 角色 speakingStyle 难以准确在 prompt 中体现 | 中 | 通过 NPC 每条消息的人设约束实现，不要求完美，有辨识度即可 |

**信心等级**：中等（IP 机制本身清晰，主要不确定性在 AI 生成角色信息的质量一致性）

## 方向二：抓马剧情怎么落地

### 2.1 抓马的本质是什么

老板问"抓马具体怎么做"，核心困惑在于：抓马不像 IP 有现成内容可以引用，也不像职场有固定场景框架。

**解答：抓马 = 高冲突情境模板 + AI 即时生成对话**

不需要预写剧本，需要预写的是"情境骨架"：
- 谁和谁有什么矛盾
- 你被推到什么位置
- 第一句话之前的紧张状态是什么

这个骨架 AI 可以基于模板即时生成，不需要人工编辑每一场戏。

### 2.2 情节模板设计：三层结构

**第一层：冲突类型（8个）**

```
1. 继承权争夺（"真假千金" 类）
2. 职场复仇（"三年后我回来了" 类）
3. 宫斗/权谋（"后宫/商战内讧" 类）
4. 秘密曝光（"你以为没人知道，但所有人都知道" 类）
5. 公开决裂（"今天必须当众说清楚" 类）
6. 盟友背叛（"我信任的人转身插刀" 类）
7. 被当众质疑（"你凭什么" 类）
8. 最后通牒（"要么你走，要么我走" 类）
```

**第二层：场景外壳（每种冲突类型对应 2-3 个外壳）**

```
冲突类型 1（继承权争夺）的外壳：
  - 董事会认亲大战（商界）
  - 遗嘱宣读现场（豪门）
  - 项目负责人争夺（职场版，门槛低）

冲突类型 4（秘密曝光）的外壳：
  - 茶会上被当众揭穿（宫斗风）
  - 公司年会上有人爆料（现代职场）
  - 发布会上竞品"意外"泄露你的数据（商战）
```

**第三层：AI 即时生成**

用户选了冲突类型（或随机），AI 基于类型和外壳生成完整会议结构：NPC 关系、关键节点、narrator 情报。用户不需要选外壳——AI 随机选一个。

**这意味着不需要预写剧本，只需要定义 8 个冲突类型的 Prompt 模板。**

### 2.3 用户在抓马场景里扮演什么角色

**答案：用户永远是"被逼到墙角"的那个人。**

不是旁观者，不是主持人——是剧情的核心受害者/反击者。这是抓马的开口动力来源：沉默等于认输，开口才有反击机会。

具体角色定位类型：

```
A. 被质疑者：你的立场/资格被当众挑战，需要反击或解释
B. 仲裁者：两方剧烈对立，都在拉你站队，你必须表态
C. 曝光者：你掌握关键信息，大家在等你说
D. 被背叛者：盟友当众转向，你需要当场回应
```

**每个关键节点的"沉默代价"要具体化**：不是抽象的"你输了"，而是 narrator 告诉你"Vivian 正在等你的反应，每多沉默 1 秒，董事们的眼神就越偏向她"。

### 2.4 抓马和 IP 能合并吗

**可以，但要谨慎。**

**合并的可行路径**："真假孙悟空大闹天庭董事会" = IP 角色（西游记）+ 抓马情节（继承权争夺）。

**但 MVP 阶段建议不合并，原因**：

1. **复杂度叠加**：合并需要用户同时做"选 IP"和"选冲突类型"两个决策，认知负担翻倍
2. **失败风险高**：IP 精准性（孙悟空说话风格是否准确）和戏剧张力（冲突是否够强）同时要保证，任何一个失败都会出戏
3. **验证混乱**：如果用户不满意，不知道是 IP 选错了还是剧情不够戏剧

**Phase 2 合并**：等 IP 模式和抓马模式分别跑通后，再做"IP x 抓马"的混搭，作为高级玩法。

narrator：对方在最后一刻加码——这是谈判常见战术，他们在赌你们需要这笔钱。

### 2.7 风险与信心等级

| 风险 | 程度 | 应对 |
|------|------|------|
| AI 生成的戏剧性不够，用户出戏 | 高 | prompt 要明确要求"NPC 对立者的立场不能软化"，narrator 要有代价感 |
| 抓马场景被玩成笑话，学习感丢失 | 中 | 复盘系统不变，语言分析深度不减；但复盘语气可以加戏剧感（"你的反击有效吗？"） |
| 8 种冲突类型 AI 实现质量参差 | 中 | MVP 先上 3 种（继承权争夺、被当众质疑、最后通牒），验证后扩展 |

**信心等级**：中等偏高（核心机制简单——高冲突 NPC + 情绪化 narrator，改动量不大；主要风险是 prompt 调优需要多轮迭代）

## 方向三：Onboarding 重新设计

### 3.1 新设计的核心逻辑

**原设计**：线性 4 步（花名 → 英语等级 → 职位 → 行业）→ 进场景选择
**新设计**：先选场景类型，根据场景类型决定后续步骤数量

**改变的原因**：
- 场景类型决定后面需要收集什么信息（IP 模式不需要职位，职场模式需要）
- 让用户一进来就感受到产品的多样性（不是又一个职场英语 App）
- 减少"不相关问题"带来的流失（IP 模式用户被问职位时会困惑）

## 方向四：用户角色分配

### 4.1 问题的本质

这个问题有两种截然不同的用户需求：

- **需求 A**：用户想当某个具体角色（"我要当孙悟空"）——有强烈的代入感驱动
- **需求 B**：用户不关心角色，只想开口练英语——任何合理角色都行

**两种需求的比例**：推测 IP 模式用户中 A 类约 40%，B 类约 60%；大乱炖用户中 A 类约 20%，B 类约 80%（大乱炖的吸引力不在于"我要当谁"而在于荒诞场景本身）。

**取舍**：两种需求都要支持，但交互设计上要让 B 类用户（不关心角色）能快速跳过。

### 4.2 角色选择的三种模式

**模式 1：自主选角色**

用户在角色展示页上看到角色卡，点击"我要扮演这个角色"。

适用场景：用户有明确偏好（"我要当孙悟空"）。

**模式 2：系统分配角色**

用户不选（点击"随机分配"或直接跳过角色选择），系统分配一个默认角色。

分配逻辑：
- IP 模式：系统优先选"旁观者"或"协调者"类型的角色（在冲突中有发言机会但不是核心对立方），降低用户的代入压力
- 大乱炖：系统分配"主持人"或"CEO"虚构身份，用户花名就是这个人的名字

**模式 3：用户扮演非 IP 角色**

无论选哪个 IP，用户可以选择"扮演一个普通与会者"——不是任何 IP 角色，而是以自己的花名参会（就像现在的职场模式一样）。

这个选项的价值：降低"要演好孙悟空"的心理压力，让用户专注于说英语而不是"角色扮演是否准确"。

### 4.3 "我要当孙悟空"之后，其他角色怎么安排

**问题**：孙悟空是强势角色，如果用户扮演孙悟空，其他 NPC 应该是谁？

**原则**：用户选了角色 X，系统从同一 IP 中选择与 X 有自然戏剧关系的角色做 NPC。

**西游记示例**：

```
用户选孙悟空 → NPC 候选：
  - 唐僧（leader/主持者，给孙悟空下达任务/施压）
  - 猪八戒（challenger，总在唱反调）
  - 沙僧（neutral/协作者，中立不表态）
  
用户选唐僧 → NPC 候选：
  - 孙悟空（challenger，激进提议）
  - 猪八戒（challenger，消极抵触）
  - 沙僧（collaborator，沉默支持）

用户选猪八戒 → NPC 候选：
  - 唐僧（leader）
  - 孙悟空（强势 challenger，压制猪八戒）
  - 沙僧（neutral）
```

这个关系映射不需要手工编写每个 IP——用 AI prompt 描述"用户选择了角色 X，请从同一 IP 中选 3 个与 X 有自然冲突关系的角色作为 NPC，说明每个 NPC 对 X 的立场（ally/neutral/pressure）"。

**大乱炖场景的角色安排**：

大乱炖里没有预设关系，NPC 关系由"角色本身的性格"决定：
- 孙悟空（急躁）天然容易和任何规则制定者（唐僧、爱因斯坦的严谨风格）产生冲突
- AI 根据角色性格 tags 推断对立/合作关系，不依赖 IP 内的原有关系

### 4.4 花名与角色选择的关系

**问题**：如果用户扮演孙悟空，花名还有意义吗？会议里 NPC 叫用户"孙悟空"还是"Alex"？

**决策**：两者共存，不冲突。

```
花名 = 会议前的账户名/昵称（UI 展示用）
角色名 = 会议中 NPC 叫用户的称呼

如果用户选了角色 → NPC 叫角色名（"Wukong"/"Sun Wukong"）
如果用户未选角色 → NPC 叫花名（"Alex"）
如果用户选了"以普通参会者身份参与" → NPC 叫花名
```

花名依然有意义：作为跨模式的身份标识，在复盘页面和历史记录中使用。会议中 NPC 的称呼只是场景内的呈现。

**Prompt 改动**：`userName` 参数改为 `speakerName`，逻辑如下：

```javascript
// generate-meeting.js 中
const speakerName = userCharacter?.name || userName;
// NPC 在 keyNode 前 cue 用户时，使用 speakerName 而不是 userName
```

### 4.5 系统默认分配逻辑（用户不选角色时）

**IP 模式默认分配**：

```
优先分配策略：
1. 如果 IP 有"协调者/顾问"类角色（如西游记沙僧、哈利波特麦格教授）→ 分配给用户
2. 如果没有，分配 ally-stance 的角色
3. 如果都不合适，分配"旁观者"虚构身份（"你是一位来访的使者/记者/顾问"）

原因：协调者/旁观者在会议中有发言机会，但不是核心对立方，
      用户不需要"演好这个角色"，只需要开口表达立场。
```

**大乱炖默认分配**：

用户扮演"会议主持人"——这是角色负担最轻的位置（主持人不需要有特定立场，只需要协调和推进）。

**抓马模式**：用户没有"选角色"的步骤——系统直接设定好用户的处境（"你是新任副总，Vivian 正在挑战你"），用户的"角色"由剧情骨架决定。

**后端（`generate-meeting.js`）**：

### 4.7 边界条件处理

**边界 1**：用户选了孙悟空，但会议主题是"年度预算规划"——孙悟空会管预算吗？

处理：systemPrompt 中明确"NPC 保持角色性格，但讨论内容服从会议主题，不需要在原著中找对应情节"。这是"平行宇宙英语版"的基本设定。

**边界 2**：用户选了 3 个都是 challenger 立场的角色（大乱炖手选）——没有盟友怎么办？

处理：角色选择页加提示"你的组合中没有盟友，会议可能很艰难——要继续吗？"或自动把其中一个改为 neutral。

**边界 3**：用户选了角色，但 Onboarding 没有填英语等级——不知道给什么难度？

处理：英语等级在场景选择后必填，不允许跳过，这个边界不存在。

### 4.8 风险与信心等级

| 风险 | 程度 | 应对 |
|------|------|------|
| "我要当孙悟空"但 NPC 没演出孙悟空的感觉，用户失望 | 高 | speakingStyle 在 NPC prompt 中要有具体约束，如"孙悟空的发言简短、直接、带轻微嘲讽" |
| 用户扮演角色后觉得"说话要像孙悟空"压力更大 | 中 | 在角色确认屏明确说明："你不需要完全模仿角色，用英语说出你的想法就好" |
| 大乱炖默认主持人身份让用户觉得太被动 | 低 | 主持人在 drama 版本里仍然是被推着发言的位置，关键节点设计不变 |

**信心等级**：中等（机制设计清晰，主要不确定性在"用户扮演角色"能否有效提升代入感，这是个需要 A/B 测试验证的用户体验假设）

## 优先级建议（基于本次分析更新）

在老板反馈约束下，建议的 MVP 路线：

**第一期**（代码改动最集中，验证最快）：

1. Onboarding 重新设计（方向三）——所有新模式的入口，必须先做
2. IP + 大乱炖合并方案（方向一）——传播性最强，用 AI 知识库实现，不需要搜索 API
3. 用户角色分配基础版（方向四）——默认系统分配，不做"用户自主选角"（Phase 2 再加）

**第二期**：

4. 抓马模式（方向二）——需要冲突模板调优，放第二期更稳
5. 用户自主选角色（方向四完整版）
6. 正常开会上传材料功能（保留现有逻辑）

**取舍显性化**：

- 抓马放第二期的代价：错过验证"戏剧张力驱动开口"这个核心假设的时机
- 抓马放第二期的理由：IP 模式可以先验证"非职场场景能否驱动开口"；抓马的 prompt 调优需要更多时间，仓促上线质量差会伤害用户信任
- IP 模式不接 web search 的代价：冷门 IP 和新番支持差
- IP 模式不接 web search 的理由：主流 IP 完全够用；搜索 API 增加成本和延迟；Phase 2 再加
